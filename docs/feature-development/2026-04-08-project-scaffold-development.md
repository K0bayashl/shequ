# 2026-04-08 - 社区 MVP 项目脚手架开发文档

> 基于 [社区项目MVP需求文档](../社区项目MVP需求文档.md) 和当前仓库工作流整理
> 文档状态：已确认
> 最后更新：2026-04-09

## 1. 基本信息

- 需求名称：社区 MVP 后端项目脚手架
- 提出人：项目负责人（个人开发者）
- 负责人：个人开发者
- 预计完成时间：待排期
- 优先级：高
- 当前状态：已确认

## 2. 背景说明

- 当前仓库已经有完整的 AI 协作工作流、记录体系和 MCP / hooks 示例，但还没有真正可运行的 Java 工程骨架。
- 在进入用户、CDK、课程、帖子等业务模块之前，需要先搭建一个稳定、可扩展、可验证的工程底座。
- 脚手架不是业务功能本身，而是后续所有业务模块的共同前提。
- 如果脚手架设计不到位，后续会在包结构、配置、测试、异常处理和构建方式上反复返工。

## 3. 脚手架应包含的内容

### 3.1 构建与运行入口

- Java 21 基线。
- Spring Boot 3.x 作为应用框架。
- Maven 作为构建工具，直接维护 `pom.xml`。
- 本地 Maven 仓库由开发者在 IDE 中配置，不强制引入 `mvnw` / `mvnw.cmd`。
- 一个可直接启动的主应用类。
- 一个最小可验证的健康检查接口或 Actuator 健康检查入口。

### 3.2 工程结构

- 按 DDD 思路预留分层目录。
- 根包下至少预留 `interfaces`、`application`、`domain`、`infrastructure`、`common`、`config` 这几类职责边界。
- 不把业务规则放进控制器，也不把数据库访问写进应用层。
- 后续业务域要能平滑扩展，而不是都堆进一个大 `service` 包。

### 3.3 基础公共能力

- 统一返回体，避免每个接口返回格式不一致。
- 统一异常处理，至少覆盖参数错误、业务错误和系统错误。
- 参数校验能力，至少支持请求对象校验和字段错误提示。
- 基础日志能力，便于排查启动问题和接口问题。
- 统一常量与错误码，避免错误消息散落在各处。

### 3.4 配置与环境管理

- `application.yml` 作为主配置入口。
- 预留 `application-local.yml`、`application-dev.yml` 等环境配置。
- 数据源、Redis、JWT 密钥等敏感信息必须支持外部化配置。
- 至少提供一份可复制的环境变量说明或示例配置说明。

### 3.5 数据库与迁移

- 接入 Flyway，保证后续数据库变更都走版本化迁移。
- 预留迁移目录和首个基线脚本。
- 在无数据库配置的本地环境下，项目也应能以本地开发模式启动。

### 3.6 安全骨架

- 预留 JWT 相关的配置、工具和过滤器入口。
- 预留密码加密组件，默认支持 bcrypt 一类的安全哈希实现。
- 默认白名单应允许健康检查和必要的开发入口，避免脚手架阶段把自己锁死。
- 当前阶段只要求搭好安全骨架，不要求把完整鉴权业务全部做完。

### 3.7 测试与质量

- 至少具备 JUnit 5 测试依赖。
- 预留 Mockito 或同类测试能力，方便后续写单元测试。
- 至少有一条能跑通的启动测试或上下文测试。
- 构建失败和测试失败要能被清楚定位。

### 3.8 文档与开发入口

- README 里要写清楚如何启动、如何测试、如何配置环境。
- 要有最小的目录说明，方便后续开发者快速找到入口。
- 如果后续接入 CI，应保证构建命令和本地命令一致。

## 4. 需求目标

- 目标 1：建立一个可运行、可测试、可继续扩展的社区 MVP 后端工程底座。
- 目标 2：让后续用户模块、CDK 模块、帖子模块和课程模块都能复用同一套工程约定。
- 目标 3：把基础设施、配置、异常、测试和迁移机制先统一起来，减少后续重复建设。
- 成功标准：项目克隆后能够直接启动，基础测试能够执行，目录结构和配置约定清晰可见。

## 5. 非目标

- 不在脚手架阶段实现用户注册、登录、CDK 激活、帖子、课程、评论等业务功能。
- 不在脚手架阶段实现前端 Vue 项目。
- 不在脚手架阶段实现完整生产部署方案、灰度发布或复杂监控体系。
- 不在脚手架阶段做大规模业务抽象，避免为了“通用”而过度设计。

## 6. 功能需求

### 6.1 项目初始化

- 生成标准 Maven 工程。
- 配置 Spring Boot 主程序入口。
- 配置基础插件与打包方式。
- 保证本地和 CI 都以 `mvn test` / `mvn spring-boot:run` 为基线命令。

### 6.2 分层目录与包结构

- 创建符合 DDD 边界的包结构。
- 预留接口层、应用层、领域层、基础设施层和公共层。
- 示例代码不要越层调用，不要把仓储和控制器混在一起。

### 6.3 基础公共组件

- 统一响应对象。
- 统一异常处理器。
- 参数校验与错误消息转换。
- 公共枚举、常量和基础工具类。

### 6.4 配置中心与环境切换

- 支持本地、开发、测试等不同环境配置。
- 支持数据库和缓存的独立配置。
- 支持敏感信息从环境变量或本地私有配置读取。
- 本地 `local` 环境在未配置数据库时也应可启动，数据库相关能力可按 profile 延后启用。

### 6.5 数据库迁移基础

- 接入 Flyway。
- 预留迁移目录。
- 通过一条基线迁移验证数据库接入是可用的。

### 6.6 安全与认证骨架

- 预留认证过滤器、JWT 工具和安全配置类。
- 预留密码编码器与当前用户上下文入口。
- 开发阶段保留白名单策略，JWT 与 Redis 只做骨架和占位配置，后续在真实业务接入时再补完整实现。

### 6.7 测试与示例接口

- 提供一个健康检查接口，作为启动验证入口。
- 提供一个最小示例接口，用于验证控制器、校验和统一返回体是否工作正常。
- 提供基础测试样例，至少能证明项目上下文可启动。
- 不要求脚手架阶段就完成 Redis、JWT、Swagger/OpenAPI 的完整功能验证。

### 6.8 文档与脚本

- README 要覆盖环境要求、启动命令、测试命令和目录说明。
- 如有需要，可补一个脚手架初始化说明，写清楚哪些文件是必须的，哪些只是模板。

## 7. 工程规则

- 领域层不依赖 Spring、HTTP、数据库 ORM 或缓存实现。
- 应用层负责用例编排，不直接写复杂业务规则。
- 基础设施层负责技术实现，不向上层泄露具体框架细节。
- 控制器只做输入输出适配，不承载核心业务决策。
- 任何新增的公共能力，都要考虑后续业务模块是否能复用。

## 8. 接口与数据

- 输入数据：项目启动参数、环境变量、配置文件、数据库连接信息。
- 输出数据：启动成功状态、健康检查响应、示例接口响应、测试结果。
- 相关接口：`GET /actuator/health` 或等价健康检查接口、最小示例接口。
- 数据库表：脚手架阶段不强制创建业务表，只要求迁移机制可用。
- 第三方依赖：Spring Boot、Maven、Flyway、JUnit 5、Mockito、MySQL、可选 Redis。

## 9. 验收标准

- 当开发者拉取脚手架后，在无数据库配置的本地环境下也可以直接启动。
- 当运行基础测试时，测试可以通过或给出明确失败原因。
- 当切换本地和开发配置时，项目可以正常读取对应环境变量。
- 当数据库配置存在时，Flyway 能顺利扫描并应用脚手架阶段的迁移脚本。
- 当访问健康检查接口时，能返回可识别的正常状态。
- 当后续开始用户模块开发时，不需要再重做工程结构、基础异常和配置体系。

## 10. 边界与异常情况

- 边界情况 1：本地没有配置数据库时，`local` 环境应仍可启动，但应明确提示数据库相关能力未启用。
- 边界情况 2：配置文件与环境变量冲突时，应有明确优先级说明。
- 异常情况 1：Flyway 迁移脚本缺失或命名错误。
- 异常情况 2：测试上下文启动失败。
- 异常情况 3：安全配置把健康检查误拦截，导致无法验证项目是否启动。

## 11. 约束条件

- 技术约束：默认后端技术栈为 Spring Boot 3.x、Java 21、Maven、MySQL 8.x。
- 时间约束：脚手架阶段优先完成最小可用底座，不做大而全的通用框架。
- 兼容性约束：后续业务模块必须能沿用当前脚手架，不要频繁改包结构和构建入口。
- 安全或合规约束：敏感配置不能硬编码，密码和密钥相关内容必须支持安全外部化。

## 12. 依赖与前置条件

- 前置需求：Java 开发环境、本地 Maven 环境。
- 外部依赖：Spring Boot 3.x、Flyway、JUnit 5、Mockito、MySQL 驱动、可选 Redis 客户端。
- 需要确认的信息：根包名、构建产物名。

## 13. 风险与待确认项

- 风险 1：如果脚手架阶段就做过多业务抽象，后续会把第一批业务开发拖慢。
- 风险 2：如果安全组件提前做得太满，可能把开发白名单、健康检查和测试入口一起拦掉。
- 风险 3：如果包结构没有提前统一，后续模块之间的边界会越来越乱。
- 待确认问题 1：根包名使用什么命名方案。
- 待确认问题 2：构建产物名使用什么命名方案。
- 已确认：脚手架当前只做后端，不同时预留前端工程入口。
- 已确认：Swagger / OpenAPI 首版只预留骨架或依赖入口，后续按实际接口开发再补。
- 已确认：Redis 和 JWT 首版只做占位骨架，不做完整业务实现。
