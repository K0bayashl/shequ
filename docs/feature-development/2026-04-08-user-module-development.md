## 1. 基本信息

- 需求名称：社区 MVP 用户相关模块
- 提出人：项目负责人（个人开发者）
- 负责人：个人开发者
- 预计完成时间：待排期
- 优先级：高
- 当前状态：已确认

## 2. 背景说明

- 业务背景：社区 MVP 的付费模式依赖用户注册、CDK 激活和会员身份识别，用户模块是所有后续课程、讨论区和后台功能的基础入口。
- 问题描述：当前仓库只有整体 MVP 需求，没有把用户模块单独整理成便于实现和拆分任务的开发文档，后续直接编码容易出现范围不清、鉴权边界遗漏和验收标准不统一的问题。
- 现有方案：总体 MVP 文档已经定义了用户角色、权限矩阵、用户系统功能和相关表结构，但还没有模块级开发文档。
- 为什么现在要做：用户模块是 MVP 第一批必须落地的核心模块，只有先完成注册、登录、身份状态识别和个人信息能力，后续内容、讨论区和后台管理才能稳定推进。

## 3. 需求目标

- 主目标：实现社区 MVP 的用户基础能力闭环，包括注册激活、登录登出、个人主页、修改密码和用户状态校验。
- 次目标：为后续课程权限、发帖权限、管理员禁用用户、管理后台用户管理等能力提供统一的用户身份与状态基础。
- 成功标准：
  - 用户可以使用有效 CDK 完成注册并直接成为激活会员。
  - 用户可以使用邮箱和密码登录，并拿到包含 `userId`、`role`、`status` 的 JWT。
  - 登录后的用户可以查看自己的个人主页信息和发布的帖子列表。
  - 用户可以在验证旧密码后修改密码。
  - 被禁用用户即使持有未过期 Token，也无法继续访问受保护接口。

## 4. 非目标

- 不做内容：
  - 不实现第三方登录。
  - 不实现邮箱验证码、短信验证码、找回密码。
  - 不实现头像上传，仅保留头像字段和展示能力。
  - 不实现管理员侧 CDK 生成能力。
  - 不实现管理员侧用户管理页面与接口。
- 暂不考虑的边界：
  - 多端同时登录管理。
  - Token 刷新机制和长期会话续签。
  - 登录风控、设备管理、异地登录提醒。
- 后续版本再做的内容：
  - 忘记密码 / 重置密码。
  - 账号安全日志。
  - 更细粒度的会员体系或购买记录。

## 5. 功能需求

- 需求 1：注册激活
  - 用户填写用户名、邮箱、密码、确认密码、CDK。
  - 系统校验用户名唯一、邮箱唯一、密码格式、两次密码一致、CDK 存在且未使用。
  - 注册成功后创建用户记录，`role=0`，`status=1`。
  - 同时将对应 CDK 标记为已使用，并记录 `used_by`、`used_at`。
- 需求 2：登录 / 登出
  - 用户使用邮箱和密码登录。
  - 登录成功返回 JWT 和必要的用户基本信息。
  - 登出阶段 MVP 只要求客户端清除 Token，后端不维护服务端会话。
- 需求 3：登录态识别与权限基础
  - 受保护接口需要从 JWT 中读取用户身份。
  - 鉴权逻辑不仅校验 Token，还要校验用户当前状态是否可用。
  - 为后续“会员可发帖 / 非会员不可看课程章节 / 管理员可进后台”提供基础用户上下文。
- 需求 4：个人主页
  - 展示用户名、头像、注册时间、会员状态。
  - 展示该用户已发布帖子列表，帖子列表可只返回基础字段。
- 需求 5：修改密码
  - 用户输入旧密码、新密码、确认新密码。
  - 系统校验旧密码正确、新密码合法且两次一致。
  - 修改成功后更新密码；是否强制重新登录由实现阶段决定，但至少要保证旧密码失效。

如果按典型流程拆分，建议按以下顺序理解：

1. 用户先获得 CDK，并在注册页填写用户名、邮箱、密码和 CDK。
2. 系统校验注册信息与 CDK 状态，成功后创建激活用户并更新 CDK。
3. 用户使用邮箱和密码登录，系统返回 JWT。
4. 用户携带 JWT 访问个人主页、修改密码等受保护接口。
5. 如果管理员后续禁用该用户，系统在鉴权阶段拒绝其后续访问。

## 6. 业务规则

- CDK 一码一用，注册成功后必须立即标记为已使用。
- 管理员账号不通过前台注册创建，由数据库初始化或后台手工创建。
- 用户注册成功后直接成为激活会员，即 `status=1`。
- `role` 与 `status` 含义如下：
  - `role=0`：普通用户
  - `role=1`：管理员
  - `status=0`：未激活
  - `status=1`：已激活
  - `status=2`：已禁用
- 后端鉴权不能只信任 JWT 里的 `status`，还要以数据库中的最新用户状态为准。
- 修改密码时必须校验旧密码，不能绕过旧密码直接更新。
- 用户名、邮箱必须全局唯一。
- 个人主页中的帖子列表只展示正常状态的帖子。

## 7. 接口与数据

- 输入数据：
  - 注册：`username`、`email`、`password`、`confirmPassword`、`cdkCode`
  - 登录：`email`、`password`
  - 修改密码：`oldPassword`、`newPassword`、`confirmNewPassword`
  - 个人主页：用户 ID 或当前登录用户上下文
- 输出数据：
  - 注册成功结果：用户基础信息、注册状态
  - 登录成功结果：JWT、用户基础信息
  - 个人主页结果：用户信息、帖子列表
  - 修改密码结果：成功 / 失败状态与提示
- 相关接口：
  - `POST /api/users/register`
  - `POST /api/users/login`
  - `POST /api/users/logout`
  - `GET /api/users/me`
  - `GET /api/users/{id}`
  - `POST /api/users/password/change`
- 数据库表：
  - `user`
  - `cdk`
  - `post`（个人主页帖子列表依赖）
- 消息事件：
  - MVP 阶段暂不引入消息事件
- 第三方依赖：
  - JWT 生成与解析组件
  - 密码加密组件（bcrypt）
  - MySQL
  - Redis 可选；如果第一阶段不做登录态缓存，可暂不接入

## 8. 验收标准

- 当用户提交合法注册信息和有效 CDK 时，应创建用户并返回成功结果，同时 CDK 状态变为已使用。
- 当用户名已存在、邮箱已存在、CDK 不存在或已使用时，注册应失败并返回明确错误提示。
- 当用户使用正确邮箱和密码登录时，应返回有效 JWT，且 Token 中包含 `userId`、`role`、`status`。
- 当邮箱或密码错误时，登录应失败并返回统一且安全的错误提示。
- 当用户携带有效 Token 访问个人主页时，应能看到自己的基本信息和帖子列表。
- 当用户未登录或 Token 无效时，访问受保护接口应返回未认证错误。
- 当用户状态为禁用时，即使 Token 未过期，也应返回无权限或账号不可用错误。
- 当用户输入正确旧密码且新密码符合规则时，应能修改密码成功，且旧密码随后不可继续登录。
- 当用户输入错误旧密码、两次新密码不一致或新密码不符合规则时，应修改失败并返回明确提示。

## 9. 边界与异常情况

- 边界情况 1：用户在同一时间重复提交注册请求，可能导致同一个 CDK 被并发消费，需要考虑原子更新或事务保护。
- 边界情况 2：用户注册时用户名和邮箱都合法，但 CDK 在校验后到提交前被其他用户先使用。
- 异常情况 1：JWT 合法但数据库中的用户已被禁用或删除。
- 异常情况 2：个人主页查询时用户存在，但其帖子列表为空。
- 异常情况 3：用户修改密码成功后，旧 Token 是否立即失效需要明确处理策略。
- 异常情况 4：管理员账号如果误走普通注册流程，应被系统阻止或在实现上避免这种入口。

## 10. 约束条件

- 技术约束：
  - 后端采用 Spring Boot 3.x。
  - 数据库采用 MySQL 8.x。
  - 鉴权采用 JWT。
  - 密码必须加密存储，不能明文保存。
- 时间约束：
  - 作为 MVP 第一阶段模块，优先实现最小闭环，不做额外安全产品化能力。
- 兼容性约束：
  - 接口返回结构应尽量稳定，便于前端后续对接。
  - 角色和状态字段命名应与总体 MVP 文档保持一致。
- 安全或合规约束：
  - 错误提示不能泄露过多账户存在性信息。
  - 密码校验和加密要遵循基础安全要求。
  - 受保护接口必须验证用户身份与状态。

## 11. 依赖与前置条件

- 前置需求：
  - Java 项目骨架已建立。
  - 用户、CDK、帖子相关数据库表已创建。
  - 基础异常处理和统一返回结构已准备好。
- 外部依赖：
  - JWT 依赖库
  - bcrypt 密码加密能力
  - 数据库连接配置
- 需要确认的信息：
  - 密码复杂度要求的最小标准
  - 修改密码后是否强制让现有 Token 失效
  - 个人主页是否允许查看他人主页，还是 MVP 只支持“我的主页”
  - 非会员 `status=0` 用户是否会在后续通过其他方式存在，还是当前阶段只会有已激活和禁用两种真实状态

## 12. 风险与待确认项

- 风险 1：如果注册与 CDK 使用状态更新不在同一事务中，可能出现用户创建成功但 CDK 未标记、或 CDK 已标记但用户创建失败的数据不一致问题。
- 风险 2：如果鉴权阶段只依赖 JWT 载荷，不回查数据库，就无法正确处理“用户已被禁用但旧 Token 仍有效”的场景。
- 待确认问题 1：个人主页帖子列表是否需要分页，MVP 第一版是否允许只返回最近若干条。
- 待确认问题 2：登录成功后是否需要返回额外的角色 / 权限信息，还是只返回基础用户信息和 JWT 即可。
