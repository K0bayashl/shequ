# AI 修 bug 提问模板

> 用于把一个 bug 描述成 AI 可以直接分析、定位和修复的问题单。建议在让 Claude 或 Copilot 修 bug 之前先填写这份模板。

## 1. 问题标题

- 问题标题：
- 问题类型：功能错误 / 回归 / 性能问题 / 兼容性问题 / 报错 / 其他
- 发现时间：
- 发现环境：本地 / 测试 / 生产 / 其他

## 2. 问题背景

说明这个问题出现在哪个功能、什么场景、影响什么用户。

- 业务背景：
- 涉及功能：
- 影响范围：
- 严重程度：高 / 中 / 低

## 3. 复现步骤

请尽量写成可以一步一步复现的过程。

1. 第一步做什么
2. 第二步做什么
3. 第三步做什么
4. 直到问题出现

如果问题不是稳定复现，也请说明：
- 偶发 / 必现
- 大约几次会出现一次
- 是否和特定数据、账号、环境有关

## 4. 期望结果

说明你原本期望系统怎么表现。

- 正确结果：
- 正确提示：
- 正确状态：

## 5. 实际结果

说明现在实际发生了什么。

- 实际表现：
- 错误提示：
- 错误状态：
- 异常日志：

## 6. 错误日志与截图

把所有能帮助定位问题的信息贴出来。

- 控制台日志：
- 后端日志：
- 浏览器控制台：
- 截图：
- 报错堆栈：
- 相关链接：

## 7. 最近改动

问题经常和最近改动有关，这部分非常重要。

- 最近改了哪些文件：
- 最近加了什么逻辑：
- 最近合并了什么分支：
- 问题是改完之后才出现的吗：是 / 否 / 不确定

## 8. 影响范围

说明问题会影响哪些功能和用户。

- 影响页面 / 接口：
- 影响用户类型：
- 是否阻塞业务：
- 是否有临时绕过方案：

## 9. 已尝试过的排查

告诉 AI 你已经做过什么，避免重复劳动。

- 已尝试方案 1：
- 已尝试方案 2：
- 已确认不是：
- 已排除的文件或模块：

## 10. 相关文件

如果你知道可能相关的代码，请直接列出来。

- 前端文件：
- 后端文件：
- 测试文件：
- 配置文件：
- 数据表 / 接口：

## 11. 修改约束

说明修复时有哪些不能碰的地方，避免 AI 乱改。

- 不能改的模块：
- 不能变的接口：
- 不能新增的依赖：
- 不能影响的行为：

## 12. 修复目标

明确告诉 AI 这次修复希望达到什么结果。

- 修复目标：
- 验收标准：
- 是否需要补测试：是 / 否
- 是否需要补文档：是 / 否

## 13. 给 AI 的输出要求

建议你在提问结尾加上这段，效果会更稳定。

- 请先分析根因，不要直接猜。
- 请给出最小修复方案。
- 请说明需要修改哪些文件。
- 请补充或更新测试。
- 请在修改后说明如何验证。
- 如果信息不够，请先问最少必要问题。

## 14. 简短示例

### 示例：保存按钮点击后报错

- 问题标题：保存按钮点击后接口返回 500
- 问题背景：用户编辑表单后点击保存，无法完成提交
- 复现步骤：
  1. 打开编辑页面
  2. 修改任意字段
  3. 点击保存
- 期望结果：保存成功并返回列表页
- 实际结果：接口返回 500，页面提示失败
- 错误日志：
  - 后端报空指针异常
- 最近改动：
  - 最近修改了校验逻辑
- 影响范围：
  - 所有编辑保存操作
- 已尝试过的排查：
  - 重试后仍然报错
  - 只有该接口受影响
- 修复目标：
  - 修复空指针并补充回归测试
