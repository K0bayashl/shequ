# Java AI 自主开发工作流

这份文档定义了一套适合从 0 到 1 开发 Java 项目的 AI 工作流。目标不是让 AI 自由发挥，而是让它按固定分工、固定验收、固定发布流程稳定产出。

## 默认技术栈

- Java 21
- Maven
- JUnit 5
- Mockito
- GitHub Actions
- 可选：Checkstyle、SpotBugs、JaCoCo

如果后续决定改用 Gradle，也可以保留同样的工作流，只替换构建与测试环节。

## 推荐的子代理

### 1. java-orchestrator

职责：接收需求、拆解任务、分派子代理、维护待办列表、控制节奏。

适用场景：
- 从需求开始做任务分解
- 协调多个子任务
- 监督实现、验证、发布流程

### 2. java-discovery

职责：只读调查仓库、查找已有模式、定位相关文件、收集外部资料。

适用场景：
- 查看项目结构
- 找到已有实现风格
- 查官方文档、库文档、API 用法

### 3. java-implementer

职责：按最小变更原则编写或修改 Java 代码，补测试，保持编译通过。

适用场景：
- 实现单个功能
- 修复 bug
- 补充单元测试或集成测试

### 4. java-reviewer

职责：审查变更，找 bug、回归风险、测试缺口、接口破坏。

适用场景：
- 代码合并前审查
- 需要第二视角判断风险
- 验证变更是否过度扩大

## 推荐的技能

### java-planning

用途：把一个需求拆成可执行步骤，明确范围、风险、验收标准和验证方式。

### java-implementation

用途：指导 AI 以最小改动实现功能，保持代码风格一致，并在修改后立刻验证。

### java-testing

用途：生成、补充、运行测试，覆盖正常路径、边界条件和回归场景。

### java-release

用途：处理 GitHub 提交、PR、发布说明、CI 结果与上线前检查。

## 推荐的 MCP

### GitHub MCP

用途：读取 issue、PR、分支、评论和仓库状态，帮助 AI 在 GitHub 上协作。

### 文档或检索 MCP

用途：查询 Java、Maven、JUnit、Spring、三方库文档，减少猜测。

### 领域 MCP

用途：如果项目连接数据库、消息队列、支付、对象存储或内部服务，就接入对应系统的 MCP。

原则：先接最常用、最稳定的 MCP，不要一开始装太多。

## 推荐的钩子

### SessionStart

作用：自动注入仓库规范、当前目标、待办清单和工作流文档。

### PreToolUse

作用：拦截高风险操作，例如危险 Git 命令、删除操作、直接发布到主分支。

### PostToolUse

作用：在修改 Java 文件后自动运行格式化、测试或轻量校验。

### PreCompact

作用：在上下文压缩前强制生成简短进度摘要，避免丢失当前状态。

## MCP 与 Hooks 落地要求

- 只要工作流里提到 MCP 或 hooks，就必须同时提供仓库内可查看的配置示例和脚本示例，不能只停留在概念层。
- MCP 相关说明优先放在 `docs/mcp-hooks-setup.md`，并注明需要安装或启用的客户端位置。
- hooks 配置优先放在 `.github/hooks/`，每个 hook 事件保留一个独立的 JSON 示例文件。
- hooks 脚本优先放在 `scripts/hooks/`，使用可直接阅读的脚本作为示例，不要只写“请自行实现”。
- 如果某个工具链因版本差异无法写出完全通用的配置，要明确标注“示例模板”，并写清楚需要替换的字段。
- 对于关键工作流（SessionStart、PreToolUse、PostToolUse、PreCompact），至少要有一种可落地的示例，避免只写“建议使用”。

## 建议的工作流

### 1. 需求接收

- 让 orchestrator 读取需求
- 明确目标、非目标、约束、验收标准
- 如果信息不足，只问一次关键问题

### 2. 仓库侦察

- 交给 discovery 子代理
- 找出相关文件、现有约定、依赖版本、测试入口
- 收集外部资料，优先官方文档

### 3. 任务拆分

- orchestrator 生成有序步骤
- 每一步只做一类变更
- 明确哪些步骤需要测试、哪些需要审查

### 4. 实现

- implementer 只改当前步骤需要的文件
- 优先小补丁，避免大范围重构
- 如果发现范围膨胀，回到 orchestrator 重新拆分

### 5. 验证

- 对应变更运行最小必要测试
- 失败时优先修复离变更最近的问题
- 不要为了通过测试顺手改无关代码

### 6. 审查

- reviewer 只看正确性、回归、接口和测试覆盖
- 输出按严重度排序
- 明确哪些地方还需要人工确认

### 7. GitHub 交付

- 在分支上完成变更
- 通过 GitHub MCP 或 Git 命令创建 PR
- PR 描述中写清楚改动、验证、风险与后续工作
- 合并前确保 CI 通过

## 自主性规则

- 没有必要就不要问用户第二次确认
- 先按现有信息推进，再在不确定处标注假设
- 每次只处理一个明确目标
- 不做与任务无关的顺手重构
- 所有公共接口变更都要更新测试和说明
- 任何修改后都要给出验证结果，或者明确说明没法验证的原因

## 推荐的交付格式

每次任务完成后，AI 至少输出：

- 本次做了什么
- 为什么这样做
- 验证了什么
- 还有什么风险
- 下一步建议
