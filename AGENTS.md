# 开发规范

本文件适用于整个仓库，AI 在本项目中进行代码阅读、设计、实现、测试和总结时都应遵守以下规则。

## 1. 总原则

- 先理解需求和现有代码，再开始修改。
- 优先做最小、最安全、最容易验证的改动。
- 不确定时先说明假设，不要默默编造。
- 任何改动都要对齐现有项目风格，不要强行统一成个人习惯。

## 2. 工作流程

- 复杂任务先拆分，再执行。
- 需要仓库侦察时，优先使用 subagent 做只读分析。
- 需要外部资料时，优先通过 MCP 获取官方文档、API 说明或其他上下文。
- 动手修改前，先明确受影响文件、风险点和验证方式。
- 修改后立即验证，避免连续引入多个未确认的变更。

## 3. 代码修改规则

- 只修改与当前任务直接相关的文件。
- 不要为了“顺手优化”而重构无关模块。
- 不要随意改名、搬文件、改接口，除非需求明确需要。
- 尽量保持原有结构、命名、缩进和格式。
- 不新增依赖，除非现有能力明显不足，并说明原因与替代方案。
- 不要加入无意义的注释；只在复杂逻辑确实需要时补充简短说明。

## 4. 测试与验证

- 任何功能修改都应补充或更新相关测试。
- 优先运行与改动最相关的单元测试、集成测试或 lint。
- 如果无法运行测试，要明确说明原因，并给出可替代的验证方式。
- 涉及 bug 修复时，验证应尽量覆盖原始问题和相邻场景。

## 5. 变更边界

- 不删除现有行为，除非需求明确要求。
- 不修复与当前任务无关的问题，除非它会直接影响本次改动。
- 影响公共接口、配置、数据结构或文档时，要同步更新调用方和说明。
- 若发现仓库中存在既有未提交改动，必须避免覆盖或回退这些改动。

## 6. 输出与沟通

- 先给结论，再给关键细节。
- 需要展示结果时，优先说明改了什么、为什么改、验证了什么、还有什么风险。
- 如果存在多个方案，优先推荐实现成本最低、风险最小、最容易维护的方案。
- 如果任务被阻塞，要明确指出阻塞点和下一步可行选择。

## 7. 优先级规则

- 用户的明确要求高于一般规范。
- 本文件高于临时习惯和默认偏好。
- 如果规范之间冲突，优先选择更安全、可验证、改动更小的方案。

## 8. 规则层级定义

为了让 AI 在每次任务开始前都能明确“要遵守什么”，本仓库把规则分成两层：

### 8.1 总规则

总规则是所有任务都必须遵守的通用约束，适用于仓库里的任何工作场景。它们主要回答“AI 在这个仓库里怎么做事”。

总规则包括：
- 先理解需求和现有代码，再开始修改。
- 优先做最小、最安全、最容易验证的改动。
- 不确定时先说明假设，不要默默编造。
- 只修改与当前任务直接相关的内容，不做无关扩展。
- 修改后必须验证，不能只改不测。
- 任务被阻塞时，要明确指出阻塞点和下一步可行选择。

### 8.2 文件级规则

文件级规则是只在某一类文件或某个具体任务范围内生效的局部约束，适用于“AI 在处理什么文件时该额外注意什么”。

文件级规则通常写在 [`.github/instructions/`](.github/instructions/) 下，并通过 `applyTo` 或 `description` 绑定到特定文件或场景，例如：
- Java 源码文件遵守 Java 代码规范。
- `pom.xml` 遵守 Maven 构建规范。
- 领域建模代码遵守 DDD 相关约定。
- 数据库迁移文件遵守 Flyway 迁移规范。

### 8.3 冲突处理

- 如果文件级规则与总规则冲突，以更安全、可验证、改动更小的方案为准。
- 如果不同文件级规则互相冲突，先停下来说明冲突，不要自行猜一个折中方案。
- 如果当前上下文不足以判断该遵守哪些规则，先提最少必要问题，再继续。

## 9. Java 项目工作流

- 在这个仓库里开发 Java 项目时，优先遵循 [docs/ai-workflow.md](docs/ai-workflow.md) 中定义的分工和流程。
- 优先使用 `.github/agents/`、`.github/skills/`、`.github/prompts/`、`.github/instructions/` 和 `.github/workflows/` 中的配置。
- 任何 Java 任务都应按“侦察 -> 规划 -> 实现 -> 测试 -> 审查 -> 发布”的顺序推进。
- 如果任务需要更高自治，先补齐子代理（subagent）和技能（skill），再扩展 hooks，而不是直接扩大主 agent 的职责。

### 9.1 Prompt-first 门禁

- 所有 Java 开发任务都必须先归类到一个明确的入口 prompt，不要直接跳进编码阶段。
- 新模块或新功能开发必须从 [.github/prompts/bootstrap-java-project.prompt.md](.github/prompts/bootstrap-java-project.prompt.md) 开始。
- 已批准的实现任务必须从 [.github/prompts/implement-java-task.prompt.md](.github/prompts/implement-java-task.prompt.md) 开始。
- bug 修复必须从 [.github/prompts/fix-bug.prompt.md](.github/prompts/fix-bug.prompt.md) 开始。
- 任务收尾必须从 [.github/prompts/close-task.prompt.md](.github/prompts/close-task.prompt.md) 开始。
- 新窗口恢复上下文必须从 [.github/prompts/initialize-window-context.prompt.md](.github/prompts/initialize-window-context.prompt.md) 开始。
- 如果当前上下文没有对应的原始文档、批准结果或任务范围，先停下来补齐信息，不要自行补全流程。
- 如果发现自己拿错了入口 prompt，先切换到正确入口，再继续后续步骤，不要一边分析一边写代码。

## 10. 项目技术栈（社区 MVP）

- 后端框架：Spring Boot 3.x。
- 数据库：MySQL 8.x。
- 缓存：Redis。
- 鉴权：JWT，Payload 包含 `userId`、`role`、`status`。
- 文件存储：阿里云 OSS。
- 前端框架：React + Next.js + TypeScript + Tailwind CSS + shadcn/ui。
- 内容格式：Markdown，支持代码块、图片和视频内嵌。
- 部署环境：阿里云 ECS。
- 会员模式：买断制社区，用户通过 CDK 激活后获得会员权限。

## 11. MCP 与 hooks 落地约定

- 仓库里凡是引用 MCP 或 hooks 的工作流，不得只写“建议”，必须同时提供可查看的配置示例、脚本示例和启用说明。
- 如果某个任务依赖 MCP 获取外部上下文，优先提供对应 MCP 的配置模板或接入说明，再让 AI 使用。
- 如果某个任务依赖 hooks 做规则确认、预检或收尾，优先提供 `.github/hooks/*.json` 示例和 `scripts/hooks/*.mjs` 示例脚本。
- 当 MCP 或 hooks 的具体客户端配置因工具版本不同而无法完全统一时，至少给出仓库内可落地的模板、目录结构和替换说明。
- 对于需要人工安装或启用的部分，要在文档中明确写出前置条件、配置位置和验证方式。
